* Goal

This tutorial runs the current research pipeline end to end:

- load the universe
- fetch asset and factor returns
- estimate forward factor models
- compute rolling and full-sample risk decompositions
- render the same decomposition plots used elsewhere in the repo

The aim is not to hide the code behind a notebook UI. It is to keep the code
visible, executable, and exportable into formats that can later feed a website
or slide deck.

* Step 2: Estimate The Forward Models

* Optional: Wrap The Pipeline In One Block

If you want one cell that behaves like =run_pipeline.py=, this block keeps the
whole workflow together while still leaving intermediate objects available in
the notebook session.

#+begin_src jupyter-python :exports code :results output
def main():
    asset_returns = get_returns(ASSETS, START, END)
    factor_returns = get_returns(FACTORS, START, END, is_factor=True)

    rolling_model_ds = rolling_forward_decompose(
        asset_da=asset_returns,
        factor_da=factor_returns,
        window=WINDOW,
        include_intercept=True,
    )

    sample_model_ds = sample_forward_decompose(
        asset_returns,
        factor_returns,
        include_intercept=True,
    )

    rolling_risk = rolling_risk_decomposition(rolling_model_ds, factor_returns)
    sample_risk = sample_risk_decomposition(sample_model_ds, factor_returns)

    print("Pipeline complete.")
    print(rolling_risk)
    print(sample_risk)

    return {
        "asset_returns": asset_returns,
        "factor_returns": factor_returns,
        "rolling_model_ds": rolling_model_ds,
        "sample_model_ds": sample_model_ds,
        "rolling_risk": rolling_risk,
        "sample_risk": sample_risk,
    }
#+end_src

* Export Targets

This Org file is useful because the same source can generate several outputs.

** HTML Report

Plain HTML is the most direct export target for a website. It preserves text,
code blocks, and image links with very little overhead.

Recommended flow:

** PDF

PDF is useful for static review copies or archival snapshots. The tradeoff is
that it is less natural for interactive browsing and linking.

** reveal.js Slides

If you want a presentation layer, Org can also export to reveal.js slides.
That is likely the cleanest path if the eventual website needs both:

- a tutorial article
- a slide-style walkthrough

The current repo already has a reveal.js example in =test_revealjs.org=. A
practical structure is:

- keep this file as the narrative report

** Website Integration

The simplest website path is to treat Org as source and exported HTML as the
published artifact.

Two workable options:

1. Export Org to HTML locally and deploy the generated files as static assets.
2. Use a static-site tool that accepts generated HTML pages under a content or
   public directory.

For this repo, the low-friction option is to export into =notebooks/reports/=
and have a website build step copy:

- =notebooks/reports/*.html=
- =notebooks/figures/*.png=

That keeps the research workflow and the publishing workflow loosely coupled.
