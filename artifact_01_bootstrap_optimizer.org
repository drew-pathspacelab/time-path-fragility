* Question

This artifact asks whether a minimum-variance portfolio is stable when the
same asset and factor history is resampled into alternative time paths.

The workflow is intentionally narrow:

- load the test asset universe and market ETF factors
- create 10 block-bootstrapped alternative paths
- estimate rolling factor decompositions on each path
- estimate factor-model covariance matrices at monthly rebalance dates
- optimize a long-only minimum-variance portfolio at each rebalance date
- summarize optimizer diagnostics and plot cross-path weight dispersion
- compare bootstrap weight and cumulative-return dispersion against the realized path

The longer exploratory notebook remains in =bootstrap_optimizer_walkthrough.org=.
This file is meant to be the cleaner export target for HTML/PDF/website work.

* Bootstrap Paths

Block resampling samples contiguous chunks of time and stitches those chunks
together into alternative paths. This keeps local return sequences intact, so
short-horizon autocorrelation, volatility clustering, and contemporaneous
asset-factor relationships are less distorted than they would be under an iid
daily bootstrap.

Shorter blocks create more path dispersion but destroy more temporal structure.
Longer blocks preserve more regime-like behavior but produce paths closer to
the realized history. A 21-trading-day block is a practical first monthly
choice for this artifact.

* Rolling Decomposition And Optimization

For each bootstrap path, this section estimates a rolling factor decomposition
and computes minimum-variance weights at monthly rebalance dates. Optimizer
diagnostics are tracked separately so failures or instability can be summarized
without interrupting the full path run.

#+begin_src jupyter-python :exports code :results none
def rolling_min_var_weights_for_path(
    asset_path_da,
    factor_path_da,
    path_id,
    tracker,
    window=WINDOW,
    rebalance_every=REBALANCE_EVERY,
    cov_shrinkage=COV_SHRINKAGE,
):
    forward_ds = rolling_forward_decompose(
        asset_path_da,
        factor_path_da,
        window=window,
        include_intercept=True,
    )
    risk_ds = rolling_risk_decomposition(forward_ds, factor_path_da)

    rebalance_times = asset_path_da.time.values[window - 1 :: rebalance_every]
    weights_by_time = []

    for t in rebalance_times:
        current_time = pd.Timestamp(t)
        start_time = current_time - pd.offsets.BDay(window - 1)

        forward_slice = forward_ds.sel(time=slice(start_time, current_time))
        factor_slice = factor_path_da.sel(time=slice(start_time, current_time))

        cov = compute_cov(
            forward_slice,
            factor_returns=factor_slice,
            method="factor_model",
            shrinkage=cov_shrinkage,
            shrinkage_target="diagonal",
        )

        weights, record = optimize(
            cov=cov,
            objective="min_variance",
            long_only=True,
            budget=1.0,
            return_diagnostics=True,
            diagnostic_context={
                "path_id": int(path_id),
                "time": current_time,
                "window_start": start_time,
            },
            raise_on_failure=False,
        )

        tracker.log(record)
        weights.name = (int(path_id), current_time)
        weights_by_time.append(weights)

    weights_df = pd.DataFrame(weights_by_time)
    weights_df.index = pd.MultiIndex.from_tuples(
        weights_df.index,
        names=["path_id", "time"],
    )

    return forward_ds, risk_ds, weights_df
#+end_src

* Diagnostics Gate

This table is a quick check before interpreting the figures. A failed optimizer
run is kept as =NaN= weights and logged in the diagnostics tables. The
bootstrap paths and realized path are summarized separately so the realized
reference run does not mask bootstrap-path instability.

#+begin_src jupyter-python :exports both :results output
def summarize_diagnostics(df):
    return pd.Series(
        {
            "n_records": len(df),
            "success_rate": df["success"].mean(),
            "constraint_violations": df["constraint_violation"].sum(),
            "median_cond_number": df["cond_number"].median(),
            "max_cond_number": df["cond_number"].max(),
            "median_obj_improvement": df["obj_improvement"].median(),
            "max_multi_init_obj_diff": df["obj_diff_init"].max(),
        }
    )

diagnostic_summary = pd.DataFrame(
    {
        "bootstrap_paths": summarize_diagnostics(diagnostics_df),
        "realized_path": summarize_diagnostics(realized_diagnostics_df),
    }
).T

print(diagnostic_summary.to_string())

* Portfolio Cumulative Return Dispersion

This figure applies the rolling optimized weights to each path's subsequent
returns. Asset returns are stored as log returns, so the helper converts them
to simple returns before aggregating across assets and compounding the
portfolio return. Weights are shifted forward one trading day so a rebalance
estimated with data through date \(t\) is applied starting after date \(t\).

#+begin_src jupyter-python :exports code :results none
def cumulative_portfolio_return_from_weights(returns_da, weights_df):
    returns_df = returns_da.to_pandas()
    weights_df = weights_df.reindex(columns=returns_df.columns)

    daily_weights = weights_df.reindex(returns_df.index).ffill().shift(1)
    valid = daily_weights.notna().all(axis=1)

    simple_returns = np.expm1(returns_df.loc[valid])
    daily_weights = daily_weights.loc[valid]

    portfolio_simple_returns = (simple_returns * daily_weights).sum(axis=1)
    cumulative = (1.0 + portfolio_simple_returns).cumprod() - 1.0
    return cumulative * 100.0

* Interpretation Notes

Read this artifact in the following order:

- first confirm the diagnostics gate has a high success rate and no repeated
  constraint violations
- then inspect whether the realized weight lines sit inside or outside the
  bootstrap percentile bands
- finally compare the timing of wide percentile bands with spikes in
  cross-path standard deviation and cumulative-return dispersion

If the rolling bands are narrow, the optimizer is stable under this bootstrap
design. If bands widen at specific dates or for specific assets, those are good
candidate periods for deeper decomposition and diagnostics in the lab notebook.
