* Bootstrap 10 Alternative Paths

Block resampling is used because it preserves short-run time structure that would be destroyed by sampling individual days independently. By resampling contiguous chunks of time, the
bootstrap keeps together local sequences of returns, which helps retain short-horizon autocorrelation, volatility clustering, and the contemporaneous joint behavior of assets and factors
within each sampled block. Because the same sampled indices are applied jointly to both asset and factor panels, cross-sectional factor correlations and the factor-residual alignment from
the original history are preserved inside each block.

This is important for the current project because the goal is not to generate completely synthetic returns, but to create alternative time paths that remain structurally consistent with
the realized market data. A simple iid bootstrap would scramble temporal dependence too aggressively and produce paths that are less faithful to the original factor-model setting.

Block size controls the tradeoff between realism and perturbation strength. Shorter blocks produce more varied paths because they break the historical sequence more aggressively, but they
preserve less local temporal structure. Longer blocks preserve more persistence and regime-like behavior, but the bootstrap paths become closer to the original realized history and
therefore show less dispersion. In practice, shorter blocks tend to expose more optimizer fragility, while longer blocks tend to make the re-estimated portfolios look more stable. The selected =BLOCK_SIZE=21=
preserves the short-horizon realism.

** Estimate Covariance and Expected Returns for Path:0

The optimizer estimates expected returns mu and covariance cov from the selected path. These estimates can be noisy, so the current implementation applies shrinkage and a small
numerical stabilization.

For expected returns, the raw estimate is the per-asset mean return from the current path. Mean shrinkage blends that estimate with a simpler target:
mu_shrunk = (1 - alpha) * raw_mu + alpha * target_mu
If the target is zero, all assets are pulled toward zero expected return. If the target is grand_mean, all assets are pulled toward the same common average return across assets in the
current estimate. This reduces sensitivity to noisy cross-sectional differences.

For covariance, the raw estimate is either the historical covariance, EWMA covariance, or factor-model covariance. Covariance shrinkage blends that raw matrix with a structured target:
cov_shrunk = (1 - alpha) * raw_cov + alpha * target_cov
A diagonal target keeps each asset’s variance but removes correlations. An identity target goes further by replacing the covariance matrix with one that has the same average variance on
the diagonal and zero correlations.

After shrinkage, the covariance matrix is stabilized numerically by forcing symmetry and adding a very small ridge term to the diagonal. Shrinkage is a statistical regularization step,
while stabilization is a numerical one. The goal is to make the optimizer less sensitive to estimation noise and less likely to fail on poorly conditioned covariance matrices.

  avg_var_hist = np.trace(cov_hist.values) / cov_hist.shape[0]
  avg_var_factor = np.trace(cov_factor.values) / cov_factor.shape[0]

  hist_identity_like = np.eye(cov_hist.shape[0]) * avg_var_hist
  factor_identity_like = np.eye(cov_factor.shape[0]) * avg_var_factor

  print("Hist cov close to scaled identity:",
        np.allclose(cov_hist.values, hist_identity_like, atol=1e-6))
  print("Factor cov close to scaled identity:",
        np.allclose(cov_factor.values, factor_identity_like, atol=1e-6))
  print()

  ones_hist = np.ones(len(cov_hist))
  closed_form_hist = np.linalg.solve(cov_hist.values, ones_hist)

* Helper Functions

#+begin_src jupyter-python :exports code :results none
def sample_min_var_weights(asset_path_da, factor_path_da, shrinkage=0.10):
    forward_ds = sample_forward_decompose(
        asset_path_da,
        factor_path_da,
        include_intercept=True,
    )

    risk_ds = sample_risk_decomposition(forward_ds, factor_path_da)
    cov = compute_cov(
        forward_ds,
        factor_returns=factor_path_da,
        method="factor_model",
        shrinkage=shrinkage,
    )
    weights = optimize(
        cov=cov,
        objective="min_variance",
        long_only=True,
        budget=1.0,
    )

    return forward_ds, risk_ds, cov, weights


def rolling_min_var_weights(
    asset_path_da,
    factor_path_da,
    window=252,
    rebalance_every=21,
    shrinkage=0.10,
):
    forward_ds = rolling_forward_decompose(
        asset_path_da,
        factor_path_da,
        window=window,
        include_intercept=True,
    )

    risk_ds = rolling_risk_decomposition(forward_ds, factor_path_da)

    rebalance_times = asset_path_da.time.values[window - 1 :: rebalance_every]
    weights_by_time = []

    for t in rebalance_times:
        current_time = pd.Timestamp(t)
        start_time = current_time - pd.offsets.BDay(window - 1)

        forward_slice = forward_ds.sel(time=slice(start_time, current_time))
        factor_slice = factor_path_da.sel(time=slice(start_time, current_time))

        cov = compute_cov(
            forward_slice,
            factor_returns=factor_slice,
            method="factor_model",
            shrinkage=shrinkage,
        )

        w = optimize(
            cov=cov,
            objective="min_variance",
            long_only=True,
            budget=1.0,
        )

        w.name = current_time
        weights_by_time.append(w)

    weights_df = pd.DataFrame(weights_by_time)
    weights_df.index.name = "time"

    return forward_ds, risk_ds, weights_df

* Notes

- The sample weight boxplot is the simplest starting visualization for
  dispersion because it keeps the path count visible.
- The heatmap is useful once you want to compare whole-path allocation patterns.
- For rolling portfolios, plotting the cross-path standard deviation at each
  rebalance date is a compact way to see *when* the optimizer becomes unstable.
- If you later want a more explicit fragility view, a good next chart would be
  a path-by-time heatmap for one asset at a time, or a stacked small-multiple
  chart of rolling median weight with 10th/90th percentile bands.
